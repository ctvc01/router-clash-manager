# 项目稳定性评估报告

**评估日期**: 2026-06-19  
**评估范围**: 架构、持久化、容错、恢复能力

---

## 核心问题清单

### 🔴 高危险等级 (影响可用性)

#### 1. **SSH连接无重试机制** 
- **现象**: 网络波动→命令失败→规则注入失败→加速失效
- **根因**: `SshService.runRemoteCommand()` 单次执行，无重试
- **影响**: 网络不稳定环境下规则更新失败
- **修复成本**: 低 (添加指数退避重试)

#### 2. **配置更新无原子性和回滚**
- **现象**: `RulesEngine.updateClashRules()` 中途失败后，配置文件处于不一致状态
- **场景**: sed命令执行到一半，网络中断 → config.yaml损坏
- **根因**: 步骤串行执行，无事务机制
- **修复成本**: 中 (引入staging区域)

#### 3. **数据文件无版本控制和增量备份**
- **现象**: 文件损坏只能重置，无法恢复历史版本
- **场景**: device_custom.json被覆写→丢失所有自定义设备名
- **根因**: `PersistenceService` 只备份不存储历史
- **修复成本**: 中 (添加时间戳备份和轮转策略)

#### 4. **Clash API调用无超时和断路器**
- **现象**: 路由器或Clash服务卡顿 → 后端挂死等待
- **根因**: `ClashApiProxy.fetchViaSSH()` 无超时上限
- **影响**: 级联超时，整个后端不响应
- **修复成本**: 低 (添加timeout参数)

#### 5. **订阅更新无校验和完整性检查**
- **现象**: 下载损坏的proxy-provider配置 → Clash核心加载失败
- **根因**: 接受任何HTTP响应，无SHA验证
- **影响**: 节点列表清空，加速完全失效
- **修复成本**: 低-中 (添加内容hash验证)

---

### 🟠 中危险等级 (影响稳定性)

#### 6. **无心跳检测，连接断裂不知道**
- **现象**: 路由器网络临时中断 → 5分钟内无感知
- **根因**: 只有请求时才检测连接，被动式检测
- **修复成本**: 低 (定期health check, 30s间隔)

#### 7. **路由器重启后无预检查**
- **现象**: 重启中Clash还未启动 → 前端请求全失败
- **根因**: `ProxyHealthService` 启动后立即测试，无预连接等待
- **修复成本**: 低 (添加15s初始化缓冲)

#### 8. **没有配置变更追踪日志**
- **现象**: 3小时前的配置改动丢失，无法追溯
- **根因**: 无变更记录，只有滚动日志
- **修复成本**: 中 (添加changelog文件)

#### 9. **日志策略不明确，长时间运行爆炸**
- **现象**: 运行30天后日志文件GB级别，磁盘告急
- **根因**: 无日志轮转(rotation)和保留策略
- **修复成本**: 低 (添加日志文件大小/时间轮转)

#### 10. **配置灰度切换不存在**
- **现象**: 规则改动直接应用，出错就整体不工作
- **根因**: 无staging环境，无验证→应用流程
- **修复成本**: 高 (需要校验框架)

## 改进优先级建议

### **阶段1: 快速增强 (1-2天, 零风险)**
这些改进不影响性能，直接提升可靠性

1. **SSH连接重试** (优先级: 最高)
   - 模式: 指数退避 (100ms → 300ms → 900ms)
   - 最多3次重试，失败后记录错误日志
   - 预期效果: 网络抖动时自动恢复

2. **API调用超时** (优先级: 最高)
   - 给ClashApiProxy所有请求加5s超时
   - 防止单个慢请求卡住整个后端

3. **心跳检测** (优先级: 高)
   - 后台每30s检查一次连接
   - 连续失败3次报警日志

4. **日志轮转** (优先级: 高)
   - 按大小轮转: 每10MB或日期回滚
   - 保留最近5个版本

5. **数据备份版本管理** (优先级: 高)
   - 设备配置改动时自动备份 (device_custom.json.backup.{timestamp})
   - 保留最近10个版本，新增恢复API

### **阶段2: 配置稳定性 (2-3天)**
这些改进确保配置变更不会造成故障

6. **配置变更日志** (优先级: 高)
   - 记录每次规则更新: 时间、设备MAC、结果、错误
   - 便于追溯和恢复

7. **配置灰度验证** (优先级: 中)
   - 生成规则前先在staging验证YAML语法
   - 校验规则是否会导致all-proxy清空

8. **故障降级方案** (优先级: 中)
   - 订阅更新失败时，使用上次缓存的节点列表
   - 不会导致节点完全消失

### **阶段3: 架构增强 (3-5天)**
这些改进提升整体可观测性和恢复能力

9. **连接池和断路器** (优先级: 中)
   - SSH连接复用，避免频繁建立
   - 连续失败>3次，断路器打开，30s后半开

10. **健康度评分系统** (优先级: 低)
    - 给路由器、Clash、订阅各打分
    - 整体评分<70分时，触发警报和自救

## 实施建议

### 快速赢利(最先做这些)

```javascript
// 改进1: SSH自动重试
const executeWithRetry = (cmd, maxRetries = 3) => {
  let lastError;
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await runRemoteCommand(cmd);
    } catch (e) {
      lastError = e;
      const delay = Math.min(100 * Math.pow(3, i), 5000);
      Logger.debug('SSH', `重试 ${i+1}/${maxRetries}，等待${delay}ms后重试`);
      await sleep(delay);
    }
  }
  throw lastError;
};
```
