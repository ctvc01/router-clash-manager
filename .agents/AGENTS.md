# 🛰️ 智能硬件与代理系统开发防灾守则

本守则提炼自系统开发过程中遭遇的路由器宕机、局域网设备断网、进程自锁等实机事故，是后续所有维护和迭代中 Agent 必须强制遵守的行为红线。

---

## 1. 🛡️ 系统安全性：防爆防宕机红线 (Anti-Explosion & Anti-Crash)
* **严禁全局重置网络**：在路由器或嵌入式系统上，**绝对禁止**运行全局清空或重启防火墙/网络服务的指令（如 `iptables -F`，`/etc/init.d/firewall restart`，`/etc/init.d/network restart` 等）。任何调整必须仅针对特定规则链，或调用系统/官方配套的细粒度局部脚本（如 `setup_iptables.sh`）。
* **切断信号联动**：凡是通过 SSH 或 Expect 启动、重启常驻后台服务（如 Clash、mihomo 等）的指令，**必须**包装在子 Shell 中并进行输入输出重定向保护，切断与 SSH 会话的信号绑定（防止 SSH 退出时发送 `SIGHUP` 信号杀掉子进程）：
  ```bash
  ( /tmp/ShellCrash/mihomo -d /data/ShellCrash -f /data/ShellCrash/config.yaml </dev/null >/dev/null 2>/dev/null & )
  ```
* **避免端口冲突与死锁**：在重启代理或重载服务时，必须在前置步骤中加入端口占用检测或缓冲延迟（如 `sleep 1.5`），防止前序进程未退出导致新进程报 `address already in use` 启动失败，造成代理与 DNS 彻底假死。

---

## 2. 📡 接口与联动：网络易失性红线 (Network Ephemerality & Timeout)
* **硬性超时与安全捕获**：网络连接总是会抖动的。任何通过 SSH 执行远程指令或发起 HTTP (Axios) 请求的代码，**必须**设置明确的超时上线限制（如 `timeout: 5000`），并且必须包裹在 `try-catch` 中进行异常兜底捕获。**绝对禁止**任何未经安全捕获的异步裸跑行为，防止链路阻断直接导致主 Node.js 进程崩退。
* **重试与退避机制**：对于 SSH 等偶发性、由于超载被 Dropbear 拒绝重置的连接（Exit Code 255 / Connection Reset），必须在连接模块中实现自适应指数退避重试（如退避 500ms 重试，最多重试 3-5 次），保障网络高负荷期间的健壮性。

---

## 3. ⏱️ 并发防拥堵：内存缓存红线 (Concurrency Control & Caching)
* **内存缓存屏障**：对于任何高频轮询的慢速设备或路由器状态查询接口（如每 10-15 秒查询一次的设备列表），**必须**引入后端的短时内存缓存（TTL 10~15秒），实现“读缓存，写失效”。**绝对禁止**让前端的轮询流量无阻拦地直达 SSH 物理执行层，防止路由器 CPU 过载。
* **串行锁与防抖**：对可能引发系统变动的高频操作（如快速点击切换设备代理模式），必须在接口端引入排队串行锁或防抖（Debounce）设计，防止并发碰撞。

---

## 4. 📝 配置防腐：配置文件纯净红线 (Configuration Purity)
* **不带特殊注释**：在任何会被上游或第三方脚本进行正则过滤、转译或生成的配置文件（如 `.list`，`.yaml`，`.conf`，`.list`）中，**严禁**写入带有空格、斜杠（`/`）或非标准字符的行内注释。
* **避免语法雪崩**：不规范的注释极易导致 YAML 转换器报错（如生成无效域名 `- ''` 导致 Clash 瘫痪），并触发错误锁定文件（如 `.start_error`）造成自愈无限死锁。

---

## 5. 🐧 跨平台兼容：极简运行红线 (Compatibility)
* **POSIX sh 兼容性**：所有编写的项目辅助脚本、检测脚本（如 `scripts/` 下的工具），必须优先写为标准的 POSIX sh 兼容脚本（即使用 `#!/bin/sh`）。
* **免除 Bash 依赖**：**禁止**使用 Bash-only 语法（如数组 `+=` 拼接、扩展括号 `{}` 等），以确保脚本在 Alpine 极简容器、BusyBox 及嵌入式 OpenWrt 终端等未安装完整 Bash 的环境里均能无障碍一键运行。
