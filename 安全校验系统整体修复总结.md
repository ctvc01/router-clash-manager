## 根本原因分析

**安全校验架构**:
```javascript
// validators.js 中的两层校验：
1. 黑名单 (FORBIDDEN_PATTERNS)
   ├─ /\brm\s+-rf\s+\// (rm -rf 根目录)
   ├─ 删除关键系统目录的 rm 操作
   └─ chmod, chown, dd, mkfs, reboot, shutdown 等

2. 白名单 (ALLOWED_COMMANDS)
   └─ 允许执行的命令列表 ← ⚠️ 不完整导致问题
```

**缺失的命令**:
- 规则处理：`sed`（修改配置文件）
- 临时文件：`rm`（清理临时文件）
- 日志查询：`tail`, `head`（日志分析）
- 字符处理：`awk`, `cut`, `tr`（字符串处理）
- 文件查询：`find`（日志清理）
- 磁盘监控：`df`（空间检查）

## 修复方案

### 提交 1: 优化黑名单规则 (a49f224)
**问题**: 黑名单过于宽泛，`/\brm\s+/` 拦截所有 rm 操作

**修复**:
```javascript
// 改为针对性拦截
/\brm\s+(-[a-z]*\s+)*\/(data|etc|root|sys|usr|lib|bin|sbin|boot|proc|var)/

// 效果：
✅ 允许 rm -f /tmp/...        （临时文件清理）
❌ 拦截 rm -rf /data/...      （危险操作）
```

### 提交 2: 精准规则 (c46c114)
**问题**: 黑名单包含 `/dev`，导致 `2>/dev/null` 被拦截

**修复**:
- 移除 `/dev` 从保护目录列表
- 理由：访问 `/dev/null` 仅用于日志丢弃，完全安全

### 提交 3: 添加 sed (7bbd729)
**问题**: 规则引擎需要 sed 处理配置文件，但被拦截

**修复**:
```javascript
ALLOWED_COMMANDS.push('sed');  // sed -i 针对特定标记段修改配置
```

### 提交 4: 全面补充 (03a7945)
**问题**: 系统审查发现其他 6 个常用命令也缺失

**修复**:
```javascript
ALLOWED_COMMANDS.push(
    'tail', 'head',    // 日志和输出处理
    'awk', 'cut', 'tr', // 字符串处理
    'find', 'df'        // 文件和磁盘查询
);
```

## 技术亮点

### 1. 黑名单策略的改进
- **从**：宽泛的 `/\brm\s+/` 匹配
- **到**：精确的 `/\brm\s+.*\/(critical-dirs)/` 匹配
- **原理**：只拦截真正危险的操作，允许安全的文件清理

### 2. 层级化安全模型
```
第 1 层：黑名单      拦截已知危险操作 (chmod, chown, rm -rf /)
   ↓
第 2 层：白名单      只允许必要命令
   ↓
第 3 层：实际执行    使用 execFile（避免 shell 注入）
```

### 3. 系统审查方法论
- 搜索所有 `runRemoteCommand()` 调用
- 提取实际执行的命令
- 对比白名单找出缺失项
- 一次性补充所有缺失命令
